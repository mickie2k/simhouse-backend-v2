บทที่ 2
ทฤษฎีและเทคโนโลยีที่เกี่ยวข้อง

ในบทนี้จะกล่าวถึงความรู้และทฤษฎีและเทคโนโลยีที่เกี่ยวข้องกับการพัฒนาระบบการสร้างรายได้แบบ C2C จากเครือข่ายเชิงพื้นที่ กรณีศึกษา เครือข่ายผู้เล่นเกม  โดยทางผู้พัฒนาได้ศึกษา รวบรวม และนำความรู้เหล่านี้มาประยุกต์ใช้ในการพัฒนาระบบ ซึ่งทฤษฎีที่เกี่ยวข้องที่ได้รวบรวมไว้ดังต่อไปนี้

2.1 ทฤษฎีแนวโน้มผู้สนใจเครื่อง Simulator ประเภทรถแข่ง
2.1.2 ทฤษฎีการจูงใจ (Motivation Theory)
ทฤษฎีการจูงใจ (Self-Determination Theory: SDT) ของ Deci และ Ryan อธิบายว่า พฤติกรรมของมนุษย์ถูกขับเคลื่อนด้วยแรงจูงใจภายใน และแรงจูงใจภายนอก โดยแรงจูงใจภายในจะเกิดขึ้นเมื่อบุคคลได้รับความพึงพอใจจากการทำกิจกรรมเอง เช่น ความรู้สึกมีความสามารถ การมีอิสระในการเลือก และการมีส่วนร่วมในสังคม 
ในบริบทของเครื่องจำลองการขับรถนั้น แรงจูงใจภายในมักเกิดจากความต้องการพัฒนาทักษะการขับรถ ความท้าทายในการแข่งขัน และความรู้สึกควบคุมพาหนะได้เหมือนจริง ซึ่งเป็นปัจจัยสำคัญที่ทำให้ผู้ใช้เกิดความสนใจและอยากใช้งานอย่างต่อเนื่อง

	2.1.3 ทฤษฎีการยอมรับเทคโนโลยี (Technology Acceptance Model: TAM)
ทฤษฎี TAM ของ Davis อธิบายว่า การยอมรับเทคโนโลยีของผู้ใช้งานขึ้นอยู่กับ 2 ปัจจัยหลัก คือ
1.	การรับรู้ถึงประโยชน์ หมายถึง ผู้ใช้เชื่อว่าเทคโนโลยีจะช่วยให้การทำงานหรือกิจกรรมมีประสิทธิภาพมากขึ้น
2.	การรับรู้ถึงความง่ายในการใช้งาน หมายถึง การที่ผู้ใช้รู้สึกว่าเทคโนโลยีนั้นไม่ซับซ้อนและเรียนรู้ได้ง่าย
ซึ่งสำหรับเครื่องจำลองการขับรถ หากผู้ใช้รับรู้ว่าเครื่องจำลองสามารถช่วยฝึกทักษะการขับจริงได้ และมีระบบควบคุมที่ใช้งานง่าย เช่น พวงมาลัย แป้น ที่ตอบสนองแม่นยำ ก็จะเพิ่มแนวโน้มความสนใจและเกิดความอยากใช้งานมากขึ้น


2.2 ทฤษฎีที่ใช้ในการพัฒนาระบบ
	2.2.1 C2C Model
	C2C (Consumer-to-Consumer) Model คือรูปแบบการดำเนินธุรกิจอิเล็กทรอนิกส์ที่เปิดโอกาสให้ผู้บริโภคสามารถซื้อขายสินค้าหรือบริการระหว่างกันโดยตรง ผ่านสื่ออิเล็กทรอนิกส์หรือแพลตฟอร์มออนไลน์ โดยไม่จำเป็นต้องมีตัวกลาง เช่น ร้านค้า หรือบริษัทขนาดใหญ่ แพลตฟอร์มที่เป็นตัวอย่างของโมเดลนี้ ได้แก่ eBay, Shopee, Facebook 
โดยความสำคัญของ C2C Model คือ
1.	เปิดโอกาสทางธุรกิจแก่บุคคลทั่วไป
ผู้บริโภคทั่วไปสามารถเริ่มต้นขายสินค้าได้โดยไม่ต้องมีธุรกิจขนาดใหญ่หรือหน้าร้าน ทำให้เกิดการกระจายรายได้และสนับสนุนเศรษฐกิจในระดับบุคคล
2.	ลดต้นทุนและเพิ่มผลกำไร
เนื่องจากไม่ต้องผ่านตัวกลาง ผู้ขายสามารถตั้งราคาที่แข่งขันได้และยังคงมีกำไรสูงขึ้น ขณะเดียวกันผู้ซื้อก็สามารถซื้อสินค้าได้ในราคาที่ถูกลง
3.	ขยายขอบเขตการค้าในระดับสากล
การใช้เทคโนโลยีอินเทอร์เน็ตทำให้ผู้ขายสามารถเข้าถึงลูกค้าทั่วประเทศหรือทั่วโลก ส่งผลให้การซื้อขายมีความสะดวก รวดเร็ว และไร้ข้อจำกัดด้านภูมิศาสตร์

2.2.2 การออกแบบ UI/UX (User Interface / User Experience)
การออกแบบ UI/UX เป็นปัจจัยสำคัญที่ส่งผลต่อความพึงพอใจและประสบการณ์ของผู้ใช้งาน การออกแบบที่ดีจะช่วยให้ผู้ใช้งานสามารถใช้งานระบบได้ง่าย เข้าใจฟังก์ชันของเว็บแอปพลิเคชันได้รวดเร็ว และลดความสับสนในการใช้งาน
2.2.2.1การออกแบบแบบ Responsive (Responsive Design)
เว็บไซต์ต้องสามารถปรับขนาดและเลย์เอาต์ตามอุปกรณ์ที่ใช้งานได้อย่างเหมาะสม เช่น บนมือถืออาจใช้เมนูแบบ Hamburger ขณะที่บนคอมพิวเตอร์จะแสดงเมนูด้านบนเต็มหน้า การออกแบบแบบ Responsive ใช้เทคนิค CSS Media Queries, Flexbox และ Grid
2.2.2.2 หลักการด้านประสบการณ์ผู้ใช้ (User Experience Principles)
ควรลดขั้นตอนที่ซับซ้อน และออกแบบให้ผู้ใช้งานเข้าใจระบบทันที พร้อมทั้งให้ Feedback ชัดเจน เช่น เมื่อผู้ใช้งานบันทึกข้อมูลสำเร็จ ระบบจะแสดงข้อความยืนยัน
2.2.2.3 ความสอดคล้องและลำดับความสำคัญของเนื้อหา (Consistency & Visual Hierarchy)
รูปแบบของปุ่ม ฟอนต์ สี และองค์ประกอบต่าง ๆ ควรสอดคล้องกันทั่วทั้งเว็บไซต์ และจัดลำดับความสำคัญของเนื้อหาให้ผู้ใช้งานสามารถโฟกัสกับข้อมูลสำคัญก่อน เช่น หัวข้อใหญ่ > หัวข้อย่อย > เนื้อหา

2.3 เทคโนโลยีที่ใช้ในการพัฒนา Software
	2.3.1 React และเฟรมเวิร์ก Next.js (Frontend)
React เป็นไลบรารี (Library) สำหรับการพัฒนา User Interface (UI) ด้วยภาษา JavaScript ที่ได้รับความนิยมอย่างแพร่หลาย โดยมักเรียกกันว่า React.js หรือ ReactJS ส่วน Next.js เป็นเฟรมเวิร์ก (Framework) ที่พัฒนาต่อยอดจาก React เพื่อช่วยในการสร้างเว็บแอปพลิเคชันที่มีประสิทธิภาพสูง รองรับการเรนเดอร์ฝั่งเซิร์ฟเวอร์ (Server-Side Rendering: SSR) และการสร้างเว็บไซต์แบบ Static (Static Site Generation: SSG) ได้อย่างสะดวกและมีโครงสร้างที่เป็นระบบ

ประโยชน์ของ React
1. Reusability – การนำส่วนประกอบกลับมาใช้ซ้ำได้ (Component-Based Architecture)
React มีแนวคิดสำคัญคือ Component ซึ่งช่วยให้สามารถแยกส่วนต่าง ๆ ของเว็บไซต์ออกเป็นส่วนย่อย ๆ ที่ทำงานอิสระต่อกัน เช่น ปุ่มกด (Button), กล่องข้อความ (Input Box), หรือโปรไฟล์ผู้ใช้ (User Profile) เป็นต้น ข้อดีของแนวคิดนี้คือสามารถแบ่งงานให้กับนักพัฒนาหลายคนทำงานในแต่ละส่วนได้อย่างชัดเจน และยังสามารถนำ Component ที่สร้างไว้แล้วกลับมาใช้ซ้ำในหลายส่วนของเว็บไซต์ได้ เช่น Modal สำหรับยืนยันการทำรายการ ซึ่งประกอบด้วยข้อความ ปุ่มยืนยัน และปุ่มยกเลิก หากพัฒนาเป็น Component เพียงครั้งเดียว ก็สามารถนำไปใช้ซ้ำได้ในทุกหน้าของเว็บไซต์โดยไม่ต้องเขียนโค้ดใหม่

2. Fast to Learn – เรียนรู้และเริ่มต้นได้รวดเร็ว
React เป็นไลบรารีที่เน้นเฉพาะส่วนของการแสดงผล (View) ในสถาปัตยกรรมแบบ MVC (Model–View–Controller) ทำให้นักพัฒนาสามารถเริ่มต้นเรียนรู้และใช้งานได้ง่าย ใช้เวลาไม่นานในการสร้างเว็บไซต์หรือเว็บแอปพลิเคชันที่มีโครงสร้างชัดเจนและจัดการได้สะดวก อีกทั้งยังมีเอกสารและชุมชนนักพัฒนาที่กว้างขวาง ทำให้สามารถค้นหาตัวอย่างหรือแนวทางการแก้ไขปัญหาได้อย่างรวดเร็ว

3. Learn Once, Write Anywhere – เรียนรู้ครั้งเดียว เขียนได้ทุกแพลตฟอร์ม
แนวคิดของ React คือ “เรียนรู้เพียงครั้งเดียว ก็สามารถนำไปใช้พัฒนาได้หลายรูปแบบ” เนื่องจาก JavaScript มีระบบนิเวศ (Ecosystem) ที่ครอบคลุมทั้งการพัฒนา Frontend และ Backend เช่น ใช้ React สำหรับพัฒนาเว็บฝั่งผู้ใช้ และใช้ Node.js สำหรับพัฒนาฝั่งเซิร์ฟเวอร์ ซึ่งช่วยลดภาระในการเรียนรู้ภาษา
2.3.2 NestJS (Backend)
NestJS เป็นเฟรมเวิร์ก (Framework) สำหรับการพัฒนาแอปพลิเคชันฝั่งเซิร์ฟเวอร์ (Backend) บน Node.js ที่ใช้ภาษา TypeScript เป็นหลัก โดยมีโครงสร้างแบบ Modular Architecture ทำให้โค้ดมีระเบียบ เข้าใจง่าย และบำรุงรักษาได้สะดวก NestJS รองรับการพัฒนา RESTful API และ GraphQL เหมาะสำหรับการทำงานร่วมกับ React หรือ Next.js เพื่อสร้างเว็บแอปพลิเคชันแบบ Full Stack ได้อย่างมีประสิทธิภาพ อีกทั้งยังสามารถเชื่อมต่อฐานข้อมูลหลายประเภท เช่น MySQL, PostgreSQL, และ MongoDB ได้อย่างยืดหยุ่น

ประโยชน์ของ NestJS
1. โครงสร้างที่ชัดเจนและเป็นระบบ (Modular Architecture)
NestJS เป็นเฟรมเวิร์กสำหรับการพัฒนาแอปพลิเคชันฝั่งเซิร์ฟเวอร์ (Backend Framework) ทีพัฒนาด้วย Node.js และใช้ภาษา TypeScript เป็นหลัก โดยมีแนวคิดการออกแบบแบบ Modular ซึ่งช่วยให้นักพัฒนาสามารถแบ่งส่วนการทำงานของระบบออกเป็นโมดูลย่อย ๆ ได้อย่างชัดเจน ทำให้โค้ดมีระเบียบ เข้าใจง่าย และบำรุงรักษาได้สะดวก

2. รองรับสถาปัตยกรรมแบบ MVC 
NestJS ใช้แนวคิดแบบ Model–View–Controller (MVC) ซึ่งเป็นแนวทางเดียวกับเฟรมเวิร์กขนาดใหญ่ในระดับองค์กร เช่น Angular หรือ Spring Boot ช่วยให้การจัดการความสัมพันธ์ระหว่างคลาสและบริการต่าง ๆ ภายในระบบเป็นไปอย่างยืดหยุ่นและมีประสิทธิภาพ

3. รองรับการพัฒนา RESTful API และ GraphQL อย่างครบวงจร
NestJS ออกแบบมาเพื่อรองรับการสร้าง RESTful API ได้อย่างมีมาตรฐาน และยังสามารถใช้ร่วมกับ GraphQL ได้ในตัว ช่วยให้การสื่อสารระหว่าง Frontend (เช่น React หรือ Next.js) และ Backend เป็นไปอย่างรวดเร็วและมีโครงสร้างชัดเจน

4.รองรับการทำงานร่วมกับฐานข้อมูลหลายประเภท
NestJS สามารถเชื่อมต่อกับฐานข้อมูลได้หลากหลาย ไม่ว่าจะเป็นฐานข้อมูลเชิงสัมพันธ์ เช่น MySQL, PostgreSQL หรือฐานข้อมูลแบบ NoSQL เช่น MongoDB ผ่าน ORM เช่น TypeORM หรือ Prisma ทำให้สะดวกต่อการจัดการข้อมูลในระบบ


2.3.3 Postgresql และ MongoDB (Database)
Postgresql เป็นระบบจัดการฐานข้อมูลเชิงสัมพันธ์ (Relational Database Management System: RDBMS) ที่ได้รับความนิยมสูง ใช้ภาษา SQL (Structured Query Language) ในการจัดการข้อมูล มีจุดเด่นด้านความเสถียร ความปลอดภัย และความสามารถในการจัดการข้อมูลที่มีความสัมพันธ์กัน เช่น ตารางลูกค้า ตารางการสั่งซื้อ และตารางสินค้า จึงเหมาะสำหรับระบบที่ต้องการโครงสร้างข้อมูลแบบตายตัวและมีการอ้างอิงระหว่างข้อมูล
MongoDB เป็นฐานข้อมูลแบบไม่เชิงสัมพันธ์ (NoSQL Database) ที่จัดเก็บข้อมูลในรูปแบบ เอกสาร (Document) โดยใช้โครงสร้าง JSON ทำให้มีความยืดหยุ่นสูง สามารถเก็บข้อมูลที่ไม่เป็นระเบียบหรือมีโครงสร้างแตกต่างกันได้ เหมาะสำหรับระบบที่ต้องการการขยายตัวรวดเร็ว เช่น แอปพลิเคชันเว็บและมือถือ

ประโยชน์ของ Postgresql และ MongoDB
1.	Postgresql – ฐานข้อมูลเชิงสัมพันธ์
ความเสถียรและปลอดภัย : เหมาะกับระบบที่ต้องการความถูกต้องและการอ้างอิงระหว่างตาราง
รองรับการทำธุรกรรม (Transactions) : ช่วยให้การจัดการข้อมูลสำคัญ เช่น การเงิน หรือคำสั่งซื้อ มีความน่าเชื่อถือ
โครงสร้างชัดเจน: การออกแบบตารางและความสัมพันธ์ช่วยให้ระบบมีมาตรฐานและง่ายต่อการบำรุงรักษา

2.	MongoDB – ฐานข้อมูลไม่เชิงสัมพันธ์
ความยืดหยุ่นสูง : รองรับข้อมูลที่มีโครงสร้างไม่ตายตัวหรือแตกต่างกันในแต่ละเอกสาร
ขยายตัวง่าย (Scalability) : เหมาะกับระบบที่มีปริมาณข้อมูลเพิ่มขึ้นอย่างรวดเร็ว
รูปแบบ JSON : ทำให้เชื่อมต่อกับแอปพลิเคชัน JavaScript ได้สะดวก เช่น NestJS, React หรือ Node.js

2.4  เครื่องมือจัดการเวอร์ชัน การทำงานร่วมกัน และการออกแบบระบบ
การพัฒนาเว็บแอปพลิเคชันจำเป็นต้องใช้เครื่องมือที่ช่วย จัดการเวอร์ชันของโค้ด การทำงานร่วมกันของทีม และการออกแบบระบบ เพื่อให้การพัฒนาเป็นระบบ มีประสิทธิภาพ และลดข้อผิดพลาด

2.4.1 การจัดการเวอร์ชันและการทำงานร่วมกัน
Git เป็นระบบ Version Control ที่ช่วยบันทึกประวัติการแก้ไขโค้ด ทำให้สามารถย้อนกลับไปยังเวอร์ชันก่อนหน้าได้ และติดตามการเปลี่ยนแปลงของไฟล์แต่ละไฟล์
GitHub, GitLab, Bitbucket เป็นแพลตฟอร์มสำหรับเก็บโค้ดและทำงานร่วมกันในทีม รองรับการสร้าง Branch, Pull Request, Merge, และ Issue Tracking ซึ่งช่วยให้ทีมสามารถพัฒนาโค้ดพร้อมกันได้อย่างเป็นระบบ

2.4.2 เครื่องมือออกแบบระบบและหน้าตาเว็บไซต์
Figma ใช้สำหรับออกแบบ Wireframe และ Mockup ของหน้าเว็บ ช่วยให้เห็นภาพโครงสร้างและหน้าตาของระบบก่อนเริ่มพัฒนา
เครื่องมือเหล่านี้ช่วยในการวาง Layout, การจัดลำดับความสำคัญของเนื้อหา และการออกแบบ UX/UI ให้สามารถใช้งานระบบได้ง่ายและเข้าใจฟังก์ชันของระบบอย่างชัดเจน

2.5 การจัดการผู้ใช้งาน
	2.5.1 การพิสูจน์ตัวตน 
การพิสูจน์ตัวตน เป็นกระบวนการสำคัญของระบบเว็บแอปพลิเคชันที่ใช้ในการ ยืนยันตัวตนของผู้ใช้งานก่อนอนุญาตให้เข้าถึงระบบ เพื่อให้มั่นใจว่าผู้ที่เข้ามาใช้งานเป็นบุคคลที่มีสิทธิ์จริง โดยทั่วไประบบจะให้ผู้ใช้ป้อนข้อมูลประจำตัว เช่น ชื่อผู้ใช้ และรหัสผ่าน เพื่อเปรียบเทียบกับข้อมูลที่จัดเก็บไว้ในฐานข้อมูล
2.5.2 การกำหนดสิทธิ์ของผู้ใช้
การกำหนดสิทธิ์ (Authorization) เป็นกระบวนการที่เกิดขึ้น หลังจากผู้ใช้ผ่านการพิสูจน์ตัวตนแล้ว เพื่อระบุว่าผู้ใช้แต่ละคนมีสิทธิ์ในการเข้าถึงข้อมูลหรือฟังก์ชันใดบ้างภายในระบบ โดยการกำหนดสิทธิ์มีความสำคัญในการจำกัดการเข้าถึงข้อมูลเฉพาะบุคคล และป้องกันการใช้งานระบบโดยไม่ได้รับอนุญาต
ตัวอย่างเช่น
1. ผู้ดูแลระบบ (Administrator) สามารถจัดการข้อมูลผู้ใช้ เพิ่ม แก้ไข หรือลบข้อมูลได้
2. ผู้ใช้งานทั่วไป (User) สามารถเข้าถึงและแก้ไขข้อมูลส่วนตัวของตนเองเท่านั้น
	2.5.3 การรักษาความปลอดภัยของข้อมูลผู้ใช้
การรักษาความปลอดภัยของข้อมูลผู้ใช้เป็นส่วนสำคัญในการพัฒนาเว็บแอปพลิเคชัน โดยเฉพาะข้อมูลที่มีความอ่อนไหว เช่น รหัสผ่านและข้อมูลส่วนตัวของผู้ใช้งาน
โดยเทคนิคที่นิยมใช้ในการป้องกันข้อมูล ได้แก่
1.Password Hashing & Salting เป็นกระบวนการเข้ารหัสรหัสผ่านก่อนเก็บลงฐานข้อมูล โดยใช้ฟังก์ชันแฮช เช่น bcrypt หรือ Argon2 เพื่อไม่ให้รหัสผ่านถูกเก็บในรูปแบบข้อความธรรมดา (Plain Text) ส่วน “Salting” คือการเพิ่มชุดอักขระแบบสุ่มเข้าไปก่อนการแฮช เพื่อป้องกันการโจมตีแบบ Dictionary Attack หรือ Rainbow Table
2.Two-Factor Authentication (2FA) เป็นการยืนยันตัวตนสองชั้น เช่น หลังจากกรอกรหัสผ่านแล้ว ผู้ใช้ต้องยืนยันผ่านรหัส OTP อีกครั้ง ทำให้ผู้ไม่หวังดีเข้าถึงบัญชีได้ยากขึ้น
3.Secure Connection (HTTPS) ใช้ใบรับรอง SSL/TLS เพื่อเข้ารหัสข้อมูลระหว่างผู้ใช้และเซิร์ฟเวอร์ ป้องกันการดักจับข้อมูล (Data Interception)

2.6 งานวิจัยที่เกี่ยวข้องกับการพัฒนาระบบ
	Chen, X., Huang, Q., & Davison, R. M. (2017). Economic and social satisfaction of buyers on consumer-to-consumer platforms: The role of relational capital. International Journal of Electronic Commerce, 21(2), 219–248.
งานวิจัยนี้มุ่งเน้นที่การทำความเข้าใจว่าปัจจัยใดบ้างที่ส่งผลต่อความพึงพอใจของผู้ซื้อบนแพลตฟอร์ม C2C โดยเฉพาะอย่างยิ่งการเน้นบทบาทของ ความสัมพันธ์ นอกเหนือไปจากปัจจัยทางเศรษฐกิจทั่วไป

วัตถุประสงค์: เพื่อพัฒนาและทดสอบโมเดลที่แสดงความสัมพันธ์ระหว่าง ทุนทางความสัมพันธ์ (Relational Capital) ของผู้ซื้อกับผู้ขาย (ซึ่งประกอบด้วยความเชื่อมั่น, ความผูกพัน, และการแบ่งปันบรรทัดฐาน) ต่อความพึงพอใจของผู้ซื้อทั้งด้าน เศรษฐกิจ (Economic Satisfaction) และ สังคม (Social Satisfaction) ในบริบทของ C2C E-commerce
ปัญหา/ช่องว่างที่ต้องการแก้ไข: งานวิจัย C2C ก่อนหน้านี้มักมุ่งเน้นไปที่ความพึงพอใจทางเศรษฐกิจจากการทำธุรกรรม (เช่น ราคาถูก, คุณภาพสินค้า) แต่ละเลย มิติทางสังคม ซึ่งมีความสำคัญอย่างยิ่งในการซื้อขายระหว่างบุคคลต่อบุคคล (P2P)
การพัฒนา: ผู้วิจัยได้เสนอโมเดลเชิงแนวคิดที่รวมเอาปัจจัยทางสังคมเข้ากับการทำธุรกรรม ดังนี้
ทุนทางความสัมพันธ์ (Relational Capital): คือความรู้สึกดีที่เกิดจากการได้รับคุณค่าที่จับต้องได้จากการทำธุรกรรม 
ความพึงพอใจทางสังคม (Social Satisfaction): คือความรู้สึกดีที่เกิดจากปฏิสัมพันธ์ทางสังคมกับผู้ขาย (เช่น การสื่อสารที่ดี, การได้รับความเคารพ, ความรู้สึกเป็นมิตร)
ความพึงพอใจทางสังคม มีผลกระทบที่ชัดเจนและแข็งแกร่งกว่าต่อ ความตั้งใจในการทำธุรกรรมซ้ำ (Repurchase Intention) ในอนาคต

จากผลการวิจัยข้างต้นชี้ให้เห็นว่า แพลตฟอร์ม C2C ควรมีการ พัฒนาระบบและฟังก์ชัน ที่ส่งเสริมปฏิสัมพันธ์ระหว่างผู้ซื้อและผู้ขาย เพื่อช่วยสร้างและรักษา ทุนทางความสัมพันธ์ เช่น ระบบการส่งข้อความโต้ตอบ ระบบการรีวิวที่เน้นคุณภาพของการสื่อสาร ไม่ใช่แค่คุณภาพสินค้า ซึ่งจะช่วยเพิ่มความภักดีของผู้ใช้ต่อแพลตฟอร์มในระยะยาว

ระบบการพัฒนาระบบการสร้างรายได้แบบ C2C จากเครือข่ายเชิงพื้นที่ กรณีศึกษา เครือข่ายผู้เล่นเกม นั้นเป็นระบบที่จะแยกออกเป็น ผู้เช่าและเจ้าของเครื่อง โดยผ่านตัวกลางคือเว็บแอปพลิเคชั่นโดยหลังจากที่ทำการ login เข้าสูระบบแล้วระบบจะมีการให้เลือกระหว่างเป็นผู้เช่าหรือเจ้าของเครื่อง โดยที่ทางผู้เช่าจะสามารถค้นหาเครื่องสำหรับเช่าเมื่อค้นหาเครื่องที่มีราคาและสเปคที่พอใจแล้วจะสามารถจองเพื่อใช้งานได้ เมื่อใช้งานจะสามารถรีวิวเพื่อเป็นการให้คะแนนได้ ทางฝั่งเจ้าของเครื่องต้องทำการลงทะเบียนเครื่องเข้าสู่ระบบ ตั้งราคา/เวลา ก่อนการปล่อยเช่าในระบบการพัฒนาระบบการสร้างรายได้แบบ C2C จากเครือข่ายเชิงพื้นที่ กรณีศึกษา เครือข่ายผู้เล่นเกม และจะสามารถดูประวัติย้อนหลังเพื่อปรับปรุงตามเหมาะสมได้

3.6	Class Diagram
 
	Class Diagram นี้ประกอบไปด้วย 22 Class โดยจะแบ่งเป็นประเภทอยู่ 3 ประเภทหลักๆคือ Controller Service และ Model โดย Controller คือ Class ที่จะทำหน้าที่เป็นรับผิดชอบจัดการคำขอและตอบกลับผู้ใช้งาน และมอบหมายงานให้ Service เป็นคนทำพวก business logic และ Model จะเป็น Class จัดการกับข้อมูลคอยสื่อสารกับ Database โดย Application นี้จะมีกลุ่ม Class อยู่ 8 กลุ่ม 
1. Auth จะเป็นกลุ่ม Class ที่คอยจัดการเกี่ยวกับ Authentication, Authorization เช่น Login, Register หรือตรวจสอบ Role ว่ามีสิทธิ์เข้าถึงหรือไม่
2. Customer เป็นกลุ่ม Class ที่จัดการเกี่ยวกับฝั่งลูกค้าเช่น ดูการจอง รีวิวการจอง เป็นต้น 
3. Host เป็นกลุ่ม Class ที่จัดการเกี่ยวกับฝั่งเจ้าของเครื่องอย่างการ อนุมัติการจอง ปฏิเสธการจอง ลงทะเบียนเครื่อง simulator เข้าสู่แอปพลิเคชัน 
4. Admin เป็นกลุ่ม Class สำหรับการจัดการเบื้องหลังของผู้ดูแลระบบที่จะสามารถ แบนผู้ใช้งานที่ผิดกฏของการใช้แอปพลิเคชัน รับรีพอร์ต อนุมัติการลงทะเบียนเครื่อง Simulator จาก Host
5. Booking เป็นกลุ่ม Class สำหรับการจองเครื่องเช่น Customer จองเครื่องเล่น A ก็จะเรียก Class Booking
6. Simulator เป็นกลุ่ม Class ที่จะจัดการเครื่องเล่น เช่น ค้นหาเครื่องเล่น ดูตารางเวลาสำหรับเครื่องเล่น
7. Schedule เป็นกลุ่ม Class ที่มีแค่ Service กับ Model จะเป็นคลาสที่จัดการเกี่ยวกับเวลาการจองเครื่องเล่นโดยจะมี class Customer กับ Simulator เรียกใช้ service นี้
8. Review เป็นกลุ่ม Class ที่มีแค่ Service กับ Model โดยจะจัดการคอมเมนต์หรือรีวิวสำหรับการจองนั้นๆ

3.7 Sequence Diagram
3.7.1 Customer 
3.7.1.1 ค้นหา
 
ผู้เช่าจะเข้าผ่านหน้าแรกของเว็บแล้วถ้ามีการกรอกสินค้าที่จะค้นหาก็จะแนบข้อมูลไปใน API ที่จะเรียก SimulatorController จากนั้นก็จะเรียก function findAll() ไปที่ SimulatorService ตัวของ SimulatorService ก็จะเรียก function สำหรับดึงข้อมูลจาก Database ผ่าน Simulator

3.7.1.2 การจอง

 

เมื่อผู้เช่าจะเข้าหน้าเว็บของเครื่องเล่นๆนั้นแล้วผู้เช่าสามารถเลือกเวลาการจองแล้วคลิกปุ่มจองจากนั้นหน้าเว็บจะเรียก API Post ไปที่ BookingController โดยแนบเวลาที่ผู้เช่าเลือกจากนั้นก็จะเรียก function สำหรับจองของ BookingService เพื่อสร้างการจองโดยจะเช็คกับ ScheduleService ก่อนว่าเวลาที่จะจองสามารถจองได้มั้ยถ้าได้ BookingService ก็จะทำการสร้าง การจองขึ้นมา แล้วจะ return Id กลับไปให้ BookingController ทำการ response ไปหน้าเว็บเพื่อแสดง การจองนั้น

3.7.1.3 รีวิว

 
เมื่อผู้เช่าใช้บริการเครื่องเล่นแล้ว ผู้เช่าสามารถรีวิวเครื่องเล่นนั้นได้ผ่านหน้าเว็บโดยจะมีช่องข้อความให้กรอก เมื่อผู้เช่าคลิกปุ่มยืนยน ข้อมูลก็จะส่งผ่าน API Post ไปที่ CustomerController โดยก็จะเรียก function review จาก BookingService แล้ว BookingService ก็จะเรียก ReviewService สำหรับรีวิวการจองนั้นเมื่อเพิ่มข้อมูลสำเร็จก็จะ return reviewId ให้แต่ละ Class จนถึง CustomerController ที่จะ response กลับไปที่หน้าเว็บว่า “Review submitted” 

3.7.2 Host
	3.7.2.1 ลงทะเบียนเครื่องเล่น

 

	ตัวของเจ้าของเครื่องจะทำการลงทะเบียนเครื่องลงในระบบโดยที่หลังจากคลิกยืนยันแล้วนั้นจะมีการส่ง API ไปยังตัวของ Hostcontroller และยืนยันตัวตนหลังจากที่ยืนยันเสร็จแล้วนั้นจะทำการสร้างไอดี และสร้างเป็นข้อมูลใหม่ขึ้นมา เพื่อบันทึกลงในระบบของแอปจากนั้นจะ return ว่าได้ทำการสร้างเรียบร้อยแล้วให้กับเจ้าของเครื่อง 






3.7.2.2 ตั้งราคา

 


การตั้งราคาและเวลาเริ่มจาก Host ตั้งราคาเวลาและส่งไปยัง HostController เพื่อบันทึกราคาและตารางเวลา จากนั้น HostController จะส่งข้อมูลไปยัง HostService ซึ่งประสานงานกับ SimulatorService เพื่ออัปเดตราคาและเวลาการใช้งาน ผ่าน ScheduleService เมื่อการตั้งค่าทั้งหมดเสร็จสมบูรณ์ ระบบจะส่งสถานะ “success” กลับตามลำดับมายัง HostController เพื่อแสดงผลลัพธ์ให้ Host ทราบว่าการตั้งราคาและเวลาสำเร็จเรียบร้อยแล้ว

3.7.2.3 ยืนยันการจอง

 

ขั้นตอนการทำงานของกระบวนการยืนยันการจอง เริ่มจาก Host (เจ้าของเครื่อง) กดปุ่ม Confirm บนหน้าเว็บ เพื่อยืนยันการจองจากนั้น Web จะส่งคำขอผ่าน API ไปยัง HostController ซึ่งเรียกเมธอด confirmBooking เพื่อให้ HostService ดำเนินการต่อ โดย HostService จะส่งต่อคำสั่งไปยัง BookingService เพื่ออัปเดตสถานะการจองในฐานข้อมูลเป็น CONFIRM เมื่ออัปเดตสำเร็จ BookingService จะส่งข้อมูลที่อัปเดตกลับไปโดยระบบจะแสดงข้อความแจ้งว่า Booking confirmed success ให้ Host เห็นบนหน้าจอเมื่อเสร็จสิ้นแล้ว
